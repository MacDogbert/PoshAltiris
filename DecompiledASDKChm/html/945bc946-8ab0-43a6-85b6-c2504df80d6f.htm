The Symantec Administrator Software Development Kit (ASDK) provides capabilities for automating and customizing Symantec Management Platform products. The Notification Server, the Site Server*, and the Symantec Management Agent** are exposed through Web services, Visual Basic Script (VBS), and a command-line interface (CLI).

*Beginning with NS 7.7, Task Services and Package Services are run on Site Servers. A Site Server can be dedicated to Task Services alone, to Package Services alone, or to both services.

**The Agent that is installed on client computers (or endpoints) is the Symantec Management Agent on Windows computers or the NS Agent for UNIX on UNIX computers.

Introduction

The Symantec ASDK provides APIs that you can use to automate and customize the Symantec Management Platform. APIs may be called through Web services, COM (e.g., for VBScript), and the Windows command line (CLI). APIs are available for the following:"

  • Notification Server
  • Task Server
  • Software Delivery Solution
  • Symantec Management Agent

To use this ASDK productively, you should read this documentation and then review the Example scenarios.

Programming Options

The ASDK offers a variety of programming options to meet the needs and skills of a diverse group of administrators. These programming options come in the form of three interface layers built in to every component of the ASDK: a Web service layer, a COM layer, and a command-line interface (CLI) layer. Each layer potentially provides a different programming language option and a different execution environment. A server execution environment means that your programs must run directly on the server computer. A remote execution environment means that your programs can run from any computer on the network and target the server computer. When determining which layer to use, consider both the programming language and the execution environment.

To help you choose your best option, the following table breaks down the three interface layers in terms of some possible programming languages, the language used for examples in the ASDK documentation, and the available execution environments.

Layer Possible Languages Language of ASDK Examples Execution Environment
Web Service C#, Visual Basic C# Server, Remote
COM Visual Basic script Visual Basic script Server, Remote
CLI CLI commands CLI commands Server

Programming Execution Environment

The execution environment determines how your program communicates with the server, which depends on which ASDK layer your program is using. Each of the three programming layers routes commands through a central ASDK layer which is housed on the server computer. (See the section How Is It Built? in Getting Started with the ASDK for further explanation.) The CLI layer communicates directly with the main ASDK layer and is therefore a Server-only option. The web service layer also communicates directly with the main ASDK layer; however, your program can communicate with the web service layer from any other computer on the network. Therefore, your program can access the web service from the server itself, or, given proper credentials, it can access the web service from a remote computer as well. The COM layer can access the main ASDK layer either through the web service layer, or directly. Most often, COM programs are routed through the web service layer, thereby providing as much remoting potential as programming with the web service layer. Alternatively, COM programs can also route commands directly to the main ASDK layer. This feature can save on overhead, but also restricts your programs to run on the server directly. Furthermore, this feature is not yet available in all ASDK components.

Further information about providing proper credentials for the web service layer and choosing local or remote routing in the COM layer are provided in the Programming Guides section.

How to Proceed

Now that you have a general understanding of what the ASDK is and what it is for, proceed in the following order:

  1. Install the ASDK.
  2. Review the ASDK architecture.
  3. Review your programming options.
  4. Review examples to get you started.
  5. Read a few helpful tips.

After you have completed those steps, you should browse the Application Programming Interfaces (APIs). The API section gives a complete overview of all of the available APIs and contains example code for each programming interface. This will be invaluable, as you write your own code, for identifying the commands the ASDK offers.

When you are ready to create you own applications, read the portions of the Programming Guides that pertain to the ASDK layer you choose to program with. This will help you start programming for the interface you choose: Web Service (C#), COM, or CLI.
